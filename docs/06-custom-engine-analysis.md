# 自研引擎深度分析

---

## 1. 好处

### 1.1 完全掌控确定性
格斗游戏对帧确定性要求极高（回滚网络依赖）。自研引擎可以：
- 从底层保证所有运算使用整数/定点数
- 不依赖第三方引擎的浮点物理系统
- 精确控制每一帧的执行顺序

### 1.2 极致性能优化
- 只实现格斗游戏需要的子系统，无多余开销
- 内存布局可针对格斗游戏数据结构优化（Cache-friendly）
- 渲染管线专为 2D 精灵批量绘制优化

### 1.3 无第三方授权风险
- 不受 Unity 授权政策变动影响
- 不依赖任何商业引擎的生命周期
- 代码完全自有，可任意修改

### 1.4 深度定制工具链
- 可以为格斗游戏专门设计帧数据编辑器
- 碰撞盒可视化编辑器完全按需定制
- 回放系统、训练模式工具可深度集成

### 1.5 技术积累
- 团队对底层有完整理解，排查问题无黑盒
- 引擎本身可复用于后续格斗游戏项目

---

## 2. 坏处

### 2.1 开发量巨大
需要从零实现的子系统（见第 3 节），每个都是独立的工程项目。

### 2.2 没有编辑器
- 无场景编辑器，所有场景数据需手写或自建工具
- 无动画编辑器，需依赖外部工具（Aseprite）并自建导入管线
- 调试工具需自建（碰撞盒可视化、帧数据查看器）

### 2.3 维护负担
- 每个平台（Windows/Mac/Linux/Web）需要单独适配
- 驱动更新、系统 API 变化需要跟进
- Bug 无社区帮助，完全靠自己

### 2.4 迭代速度慢
- 引擎 Bug 和游戏 Bug 混在一起，排查困难
- 早期大量时间花在基础设施而非游戏内容上

### 2.5 人力成本
- 专业引擎团队通常需要 5-10 人专职维护
- 独立开发者同时维护引擎和游戏内容，精力严重分散

## 3. 需要实现的子系统清单

### 3.1 渲染系统
```
渲染子系统
├── 窗口管理（创建窗口、处理分辨率）
├── 图形 API 封装（OpenGL / Vulkan / Metal / DirectX）
├── 精灵渲染（Sprite Batch，批量绘制减少 Draw Call）
├── 纹理管理（加载、缓存、Atlas 打包）
├── 动画系统（帧序列播放、帧事件回调）
├── 视差背景（多层滚动）
├── 粒子系统（命中特效、超必杀特效）
├── 后处理（可选：屏幕抖动、闪白效果）
└── 调试渲染（碰撞盒可视化、帧数据叠加显示）
```

### 3.2 输入系统
```
输入子系统
├── 键盘输入（原始按键状态）
├── 手柄输入（XInput / DirectInput / SDL GameController）
├── 输入映射（按键重绑定）
├── 输入缓冲（16 帧历史队列）
└── 指令识别（QCF/QCB/DP/HCF 等）
```

### 3.3 音频系统
```
音频子系统
├── 音效播放（WAV/OGG，低延迟）
├── 音乐播放（BGM 流式加载）
├── 音量控制（主音量/音效/音乐分离）
├── 3D 音效（可选，格斗游戏通常不需要）
└── 音频混合（多音效同时播放）
```

### 3.4 物理/碰撞系统
```
碰撞子系统（格斗游戏专用，不需要通用物理引擎）
├── AABB 碰撞检测（矩形 vs 矩形）
├── 碰撞盒类型管理（Hitbox/Hurtbox/Pushbox/Throwbox）
├── 地面检测
├── 推挤分离（Pushbox 重叠时向外推开）
└── 全程整数坐标（保证确定性）
```

### 3.5 资源管理
```
资源子系统
├── 纹理加载（PNG/BMP）
├── 音频加载（WAV/OGG）
├── 帧数据加载（JSON 解析）
├── 资源缓存（避免重复加载）
└── 资源打包（发布时打包为单一文件）
```

### 3.6 网络系统
```
网络子系统
├── UDP Socket 封装（低延迟传输）
├── 回滚网络逻辑（可直接集成 GGPO 库）
├── Save State / Load State（快照序列化）
├── 输入同步协议
└── 匹配/大厅（可对接 Nakama 或 Steam）
```

### 3.7 游戏逻辑层（格斗游戏专用）
```
游戏逻辑
├── 状态机（HFSM）
├── 帧数据系统（JSON 驱动）
├── 连招系统（取消链、伤害衰减）
├── 气槽系统
├── 摄像机系统
├── 回合管理（计时器、胜负判定）
└── UI 系统（血条、气槽、计时器 HUD）
```

## 4. 可用的基础库（封装层）

完全不需要从零写 OpenGL 调用，有大量成熟库可以封装：

### 4.1 窗口 + 输入 + 音频一体化

| 库 | 语言 | 说明 |
|----|------|------|
| **SDL3** | C | 业界标准，跨平台窗口/输入/音频，几乎所有平台支持 |
| **SFML** | C++ | 面向对象封装，比 SDL 更易用，2D 渲染内置 |
| **raylib** | C | 极简 API，学习曲线最低，适合快速原型 |
| **sokol** | C | 极轻量，单头文件，适合嵌入式或 WASM |

### 4.2 渲染专用

| 库 | 语言 | 说明 |
|----|------|------|
| **SDL_GPU**（SDL3 新增） | C | SDL3 内置现代 GPU API，跨平台 |
| **bgfx** | C++ | 跨平台渲染抽象层，支持 DX11/12/Vulkan/Metal/OpenGL |
| **wgpu**（Rust） | Rust | WebGPU 标准实现，支持 WASM，Bevy 底层使用 |
| **OpenGL + GLAD** | C/C++ | 经典方案，教程最多 |

### 4.3 音频专用

| 库 | 语言 | 说明 |
|----|------|------|
| **miniaudio** | C | 单头文件，跨平台，零依赖 |
| **FMOD** | C++ | 商业级，功能强大，免费额度 |
| **OpenAL Soft** | C | 开源，跨平台 |
| **SDL_mixer** | C | SDL 配套音频库 |

### 4.4 网络

| 库 | 语言 | 说明 |
|----|------|------|
| **GGPO** | C++ | 格斗游戏回滚网络标准库 |
| **ENet** | C | 可靠 UDP，轻量 |
| **GameNetworkingSockets** | C++ | Valve 出品，Steam 底层 |

### 4.5 工具库

| 库 | 语言 | 说明 |
|----|------|------|
| **nlohmann/json** | C++ | JSON 解析，帧数据读取 |
| **stb_image** | C | 单头文件图片加载 |
| **Dear ImGui** | C++ | 调试 UI，碰撞盒编辑器可用 |
| **EnTT** | C++ | 轻量 ECS 库，可选 |

## 5. 最小可行引擎技术栈

如果决定自研，最务实的组合：

### 方案 A：C++ + SDL3（最成熟）
```
SDL3               → 窗口、输入、基础渲染
SDL_mixer / miniaudio → 音频
GGPO               → 回滚网络
nlohmann/json      → 帧数据解析
stb_image          → 纹理加载
Dear ImGui         → 调试工具
```

### 方案 B：Rust + SDL3/winit（现代）
```
winit              → 窗口管理
wgpu               → GPU 渲染
rodio / kira       → 音频
GGRS               → 回滚网络（Rust 原生）
serde_json         → 帧数据解析
```

### 方案 C：C + raylib（最简单）
```
raylib             → 窗口/渲染/输入/音频一体化
GGPO               → 回滚网络
cJSON              → 帧数据解析
```

## 6. 现实评估

| 子系统 | 使用成熟库 | 预估工作量 |
|--------|-----------|-----------|
| 窗口 + 输入 | SDL3 / raylib | 小（1-2 周） |
| 2D 精灵渲染 | SDL3 GPU / SFML | 中（2-4 周） |
| 音频 | miniaudio | 小（1 周） |
| 碰撞检测 | 自建（AABB 简单） | 小（1 周） |
| 状态机 | 自建 | 中（2-3 周） |
| 帧数据系统 | 自建 + JSON 库 | 中（2-3 周） |
| 输入缓冲/指令识别 | 自建 | 中（2-3 周） |
| 回滚网络 | GGPO / GGRS | 大（4-8 周） |
| 调试工具 | Dear ImGui | 中（2-4 周） |
| **合计** | | **约 4-6 个月** |

> 对比：使用 Godot 4，上述大部分系统开箱即用，可节省约 3-4 个月。

---

## 7. 结论

**自研引擎适合你的条件：**
- 团队有 C++/Rust 底层经验
- 项目长期规划（多款格斗游戏复用引擎）
- 对性能/确定性有极致要求
- 愿意投入额外时间在基础设施上

**不适合的条件：**
- 团队规模小（1-3 人）
- 希望尽快出可玩版本
- 没有底层图形编程经验

**折中方案：** 使用 raylib 或 SDL3 封装一个"薄引擎层"，只实现格斗游戏需要的功能，避免通用引擎的冗余，同时大幅降低工作量。这是介于"完全自研"和"使用 Godot"之间的选择。
